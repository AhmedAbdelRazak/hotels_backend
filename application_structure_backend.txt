JANNATBOOKING BACKEND APPLICATION STRUCTURE
Audited: 2026-05-23

Purpose
This file is a compact backend reference for future development. It documents the active Express/Mongo API structure, the important route files, the user-role model, hotel scoping rules, and the workflows that must be protected when changing the system.

Runtime Entry Point
- server.js
  - Creates the Express app, HTTP server, Socket.IO server, MongoDB connection, middleware, and route mounting.
  - Loads every file in ./routes dynamically under the /api prefix.
  - Serves static uploads from /uploads.
  - Starts the housekeeping maintenance scheduler after MongoDB connects.
  - Socket rooms currently used:
    - support case rooms by support-case id.
    - housekeeping:{hotelId} for housekeeping updates.
    - hotel-notifications:{hotelId} for hotel notification counters.
    - owner-notifications:{ownerId} for owner-level notification counters.
  - Watches SupportCase changes and broadcasts the latest conversation activity.

Top-Level Backend Structure
- aiagent/
  - AI agent orchestration, learning, policy, memory, ingestion, and reservation support utilities.
- controllers/
  - Business logic for every domain: auth, users, hotels, rooms, reservations, housekeeping, overall dashboards, reports, payments, support, OTA tools, and wallet operations.
- models/
  - Mongoose schemas for users, hotels, rooms, reservations, housekeeping tasks, support cases, expenses, agent wallets, activity tracking, and integration data.
- routes/
  - Express route definitions. Most routes delegate to controllers and are loaded by server.js under /api.
- services/
  - Shared backend services for notifications, activity tracking, reservation pricing/status, OTA email mapping, and housekeeping maintenance.
- uploads/
  - Uploaded hotel images, user documents, reservation documents, and generated/imported assets.

Core User Roles And Scope Rules
- role 1000
  - Platform admin / super admin level. In business terminology, Super Admins are the platform owners/operators, and platform admins belong to the JannatBooking platform side.
  - Also treat REACT_APP_SUPER_ADMIN_ID and SUPER_ADMIN_ID style configured users as super admins where used.
  - Backend helpers must also recognize roleDescription/roleDescriptions values such as superadmin and super admin as platform-super-admin grants.
  - Can access admin routes, owner-style hotel operations, account management, hotel-staff updates, and cross-hotel reporting without being restricted by hotelIdWork, hotelIdsWork, hotelIdsOwner, hotelsToSupport, or belongsToId.
- role 2000
  - Root hotel owner when the account has no belongsToId.
  - Also appears for scoped hotel managers when roleDescription is hotelmanager and hotelIdWork / hotelIdsWork / belongsToId are set.
- role 10000
  - Hotel System Admin for an owner account. This is still a hotel-side employee/admin for one hotel or a group of hotels.
  - Broad owner-like access, but always scoped to assigned hotels through hotelIdsOwner, hotelsToSupport, hotelIdWork, or belongsToId.
  - Keep the internal roleDescription value as systemadmin for compatibility, but user-facing UI and documentation should call it Hotel System Admin, not generic System Admin.
- role 3000
  - Reception. Reservation-facing access for assigned hotel scope.
- role 4000
  - Housekeeping manager. Housekeeping reports, task assignment, and operational task management for assigned hotels.
- role 5000
  - Housekeeper. Assigned housekeeping task access.
- role 6000
  - Finance. Financial dashboards, reports, wallet/commission actions, and selected reservation reporting within hotel scope.
- role 7000
  - Order taker / external agent. Usually limited to newReservation and ownReservations access. Booking source should default to the agent/company identity.
  - Public agent signup creates this role as inactive with agentApproval.status = pending and applicationReview.status = pending until a hotel owner/manager approves it.
- role 8000
  - Reservation employee. Reservation operations, pending confirmations, and selected settings/report access.
- role 0
  - Legacy/client/public-style role still appears in older code paths and should be treated carefully before removal.

Role Data Model Notes
- models/user.js is the source of account and staff-role data.
- Important fields:
  - role: primary numeric role.
  - roleDescription: primary text role.
  - roles and roleDescriptions: additional role grants.
  - belongsToId: parent owner/account id for staff or scoped users.
  - hotelIdWork and hotelIdsWork: assigned hotel scope used by hotel staff flows.
  - hotelIdsOwner and hotelsToSupport: owner/hotel-system-admin support scope.
  - accessTo: feature-level access such as newReservation and ownReservations.
  - activeUser: account activation gate.
  - companyName, agentCommercialModel, agentApproval, wallet opening balances: agent/ordertaker finance context.
  - applicationReview: public signup review state for agent and job applicants before hotel approval.

Critical Backend Authorization Helpers
- controllers/auth.js
  - createHotelStaffUser maps role descriptions to numeric roles and validates that the creator can create accounts for the requested hotels.
  - Finance and reservation employee roles can create external agent/order-taker accounts in specific paths.
  - isPlatformSuperAdmin is the global admin source for auth middleware and staff creation. isAuth, isAdmin, isHotelOwner, agent approval, and staff-creation checks should use this broad helper instead of only checking a single configured id or role number.
- controllers/overall_dashboard.js
  - requireOverallSection, canAccessOverallSection, and getAccessibleOverallHotels are the main overall-dashboard guards.
  - Owner-like access is not the same as global access. Hotel System Admins and scoped managers must still be limited to assigned hotels.
  - Platform super admins must load overall hotels globally by default. ownerId may narrow the view, but stale hotelIdWork/hotelIdsWork/hotelIdsOwner/hotelsToSupport values on the super-admin account must never restrict global access.
  - Overall reservation list sorting supports the frontend sort controls for booking source, hotel name, arrival date, and departure date through sortFromQuery.
- controllers/reservations.js
  - canCreateReservationForActor and canViewReservationHotel protect reservation creation and visibility.
  - Order takers are restricted to their allowed reservation flows and own reservation records.
  - Reservation status progression and backward status changes must remain protected.
- controllers/user.js
  - canManageHotelStaff protects staff/account management by hotel scope.
  - isPlatformSuperAdmin must be treated as an unrestricted staff/account-management actor. Super admins can update any hotel staff or owner-like account through PUT /hotel-staff/:staffId/:hotelId/:userId.
  - updateHotelStaffUser must recognize root owner accounts by _id === ownerId and hotelIdsOwner, not only belongsToId. Root owner accounts normally have no belongsToId and should not be rejected as "not belonging to this hotel".
  - Agent and employee activation from the overall activate-accounts screen should still go through updateHotelStaffUser, not the Hotel System Admin endpoint, so activeUser, agentApproval/applicationReview, approvedHotelIds, notifications, and ActivityTracker updates remain consistent.
- controllers/agent_wallet.js
  - Financial access guards protect agent wallet, claims, transactions, and commission review.
  - Wallet claim statuses drive finance reporting: posted/approved movements are accepted and can affect wallet totals; pending/rejected/void movements must remain visible for audit but are reconciliationEligible = false and must never be included in reservation reconciliation totals. Both status and reviewStatus must be checked so mixed legacy records cannot slip into totals.

- routes/expenses.js

- Payment and payout route files

- Integration route files
  - otaInbound.js, integrator.js, and aiagent_learning.js.

Core Models
- models/user.js
  - Account, role, staff scope, agent/company, and access data.
  - notificationAcknowledgements stores per-user read receipts for generated dashboard notifications. Use stable ack keys so resolved agent-facing wallet/reservation notifications disappear after click without hiding pending operational queues.
- models/signup_invitation.js
  - Stores short public signup invitation records for /signup?invite=<code>. Codes are stored only as SHA-256 hashes; payloads include signed invitation scope such as account type, hotels, hotel names, role, agent commercial model, optional prefill data, creator, and expiry.
- models/hotel_details.js
  - Hotel profile, owner link, setup completion fields, rooms/photos/location/payment settings, activation state, and wallet/payment fields.
- models/rooms.js
  - Physical room records, room type/display, room cleanliness state, housekeeping dirty metadata, bed setup, and hotel ownership.
- models/reservations.js
  - Reservation status, guest details, confirmation number, room ids, pricing, payment data, source/order-taker fields, pending confirmation state, commission snapshots, wallet snapshots, financial cycle, and audit fields.
- models/housekeeping.js
  - Housekeeping tasks, room/task status, assignedTo/assignedBy, status history, start/end timestamps, duration, and hotel scope.
- models/agent_wallet.js
  - Agent wallet balances, claims, transactions, review/void flows, and commission tracking.
- models/expenses.js
  - Hotel/account expenses and financial-report support data.
- models/supportcase.js
  - Support cases, conversations, seen/unseen tracking, and case status.
- Supporting models
  - activity_tracker, housekeeping_supply, housekeeping_supply_request, Uncompleted, aiagent learning models, and integration/inbound data.

Important Services
- services/housekeepingMaintenance.js
  - Scheduled job that marks rooms dirty after the configured clean-room window.
- services/notificationEvents.js
  - Emits hotel and owner notification counter updates through Socket.IO rooms.
- services/reservationStatus.js
  - Shared pending-confirmation and reservation-status helpers.
- services/reservationPricing.js
  - Reservation pricing calculations and shared pricing behavior.
- services/activityTracker.js
  - Writes operational activity/audit history.
  - Tracks reservation status/scope changes, reservation exports, financial report exports, account creation, and account updates.
  - Account tracking stores safe before/after field changes for role, hotel scope, active state, access, agent wallet setup, approval status, and company-document metadata only. It must never store plaintext passwords, password hashes, or full attachment data.
- services/auditPrivacy.js
  - Central privacy guard for reservation/admin tracker visibility.
  - Reservation adminChangeLog and reservationAuditLog entries made by superadmin/admin/systemadmin actors are removed from backend responses unless the requesting viewer is a true super admin.
  - Privileged actor snapshots such as adminLastUpdatedBy, pendingConfirmation.lastUpdatedBy, agentDecisionSnapshot.decidedBy, financial_cycle.*By, and commission approval reviewers are also nulled for non-super-admin viewers so ordinary agents/staff never see platform admin names in reservation details.
  - Optional Authorization headers are respected on public-safe single-reservation fetches so super admins can see privileged tracker rows while public/client views and ordinary hotel users cannot.
  - ActivityTracker records created by privileged actors are marked visibility=super_admin_only for future reporting surfaces.
- services/otaEmailOrchestrator.js and services/otaReservationMapper.js
  - OTA email/reservation parsing and mapping support.

Critical Workflows To Protect
- Reservation creation
  - Always verify the acting user can create a reservation for the target hotel.
  - For overall-dashboard reservation creation, require the hotel dropdown to be selected from hotels assigned to the user.
  - For agents/order takers, source should come from the agent/company identity unless a role explicitly requires a manual source.
- Public signup applications
  - Direct public signup is for hotel manager/owner registration only.
  - Agent and job applicants require a valid backend invitation generated from the overall create-account flow. New invitations use a short /signup?invite=<code> URL backed by models/signup_invitation.js; legacy invitationToken links are still accepted for compatibility.
  - GET /signup-invitation/:code resolves a short code only after checking the stored hash, expiry, revoked state, and invited hotel records. It returns public-safe hotel names/locations and locked invitation fields for the frontend.
  - POST /property-listing must remain the source of truth for invitation validation, hotel validation, and pending approval creation. It accepts invitationCode first and falls back to invitationToken.
  - Signed invitations may include assigned hotels that are not public/published on the Janat frontend; public non-invited agent applications must still use active published hotels.
  - If a pending applicant requested hotels across owners, approval must scope the account to the approving owner's hotel(s), not blindly activate every requested hotel.
  - Application attachments are stored in companyDocuments and may include company files, CVs, or wallet/payment proof.
  - Emit hotel/owner notification refresh events when a public application is submitted or approved.
  - Keep applicationReview and agentApproval in user read/auth payloads because the pending applicant dashboard refreshes its approval status from /user/:userId.
  - Agent-facing notifications
  - GET /reservations/notifications/pending-confirmation/:userId generates reservation, finance, account, and wallet notification feeds. Resolved agent wallet claims and accepted/rejected reservation decisions include ackKey/ackable fields.
  - PUT /reservations/notifications/pending-confirmation/:userId/acknowledge stores the ack key on the user. Pending work-queue items should not be made ackable; resolved account approvals/rejections, wallet claim approved/rejected, and reservation/finance accepted notifications should disappear after one click.
- Reservation status changes
  - Prevent non-super-admin users from moving reservation/task status backward.
  - Closed/completed records should not be reopened except by a super admin path that also clears start/end timestamps where required by the business rule.
- Tracker privacy
  - Any reservation tracker/audit update performed by a super admin, platform admin, or Hotel System Admin must only be visible to true super admins.
  - Apply services/auditPrivacy.js before returning reservation details, pending-reservation queues, finance/payment commission rows, reservation searches, and report rows that include adminChangeLog or reservationAuditLog.
  - Sanitize both audit arrays and standalone "last updated by" snapshots before responding; hiding the table rows alone is not enough because ReservationDetail also renders workflow summary metadata.
  - Do not rely on frontend hiding alone for privileged audit rows. Backend responses should already be sanitized for non-super-admin viewers and for unauthenticated public/client payment fetches.
- Housekeeping task assignment
  - Managers and authorized hotel users can assign/update housekeeping tasks within assigned hotel scope.
  - Housekeepers must only receive and act on tasks assigned to them or scoped to their hotel.
  - Notifications must be emitted through hotel/owner/housekeeping socket rooms when assignments or task states change.
- Overall dashboard
  - Use backend hotel-scope helpers for every multi-hotel query. Never trust ownerId or hotelId query values without checking the actor's allowed hotels.
  - Platform super admins are the exception to hotel-scope limits and must remain global across ownerId/hotelId contexts.
  - Overall pending-reservations is available to approved order-taker/agent accounts as a read-only operational queue. Agents must only receive reservations tied to their own createdByUserId/orderTakeId/createdBy/orderTaker identifiers, while hotel staff keep their normal pending/finance workflow scope.
- Account management
  - Staff creation and updates must validate that the creator controls every hotel selected for the new or updated account.
  - Super admins can create, activate, update, approve, and scope accounts across any hotel. Do not add hotel assignment restrictions to role 1000 / configured super-admin / superadmin role-description users.
  - Root owner accounts should preserve hotelIdsOwner and avoid forcing belongsToId during account updates. Updating a root owner through staff-management routes must not convert it into a scoped staff account.
  - PUT /overall-dashboard/system-admin/:accountId/:userId and POST /overall-dashboard/system-admin/:userId are legacy route names for Hotel System Admin account management. They must validate hotel assignments differently by actor: super admins validate selected hotel IDs against real HotelDetails records globally; owner/hotel-system-admin/hotel-manager actors validate selected hotel IDs as a subset of their accessible overall hotels.
  - The system-admin update route may convert an existing managed account into a Hotel System Admin. It must force role/roles/roleDescription/roleDescriptions to systemadmin only, clear belongsToId, set hotelIdWork/hotelIdsWork/hotelIdsOwner/hotelsToSupport from the selected hotels, and preserve sanitized accessTo.
  - Staff creation with roleDescription systemadmin must also keep Hotel System Admin mutually exclusive, set hotelIdsWork/hotelIdsOwner/hotelsToSupport from selected hotels, and avoid assigning belongsToId like a regular hotel employee.
  - Converting a Hotel System Admin back to hotel manager/reception/finance/etc. is handled through the hotel-staff update route. That route must allow accounts scoped by hotelIdsOwner/hotelsToSupport, then clear hotelIdsOwner for non-system-admin non-root-owner accounts and restore belongsToId to the hotel owner.
  - Account create/update routes must call services/activityTracker.js. Staff creation uses hotel_staff_create, staff updates use hotel_staff_update, and Hotel System Admin create/update uses overall_system_admin_create / overall_system_admin_update.
  - Overall create-account must not auto-select hotels or roles. Backend account creation and signup invitation creation should reject missing hotel/role scope, even if the frontend already validates it.
  - Overall update-account should require an explicit account selection in the frontend before update fields are populated. Backend update routes must continue validating account id, creator scope, selected hotels, and role changes because query params or prefilled UI state are never trusted.
  - Signup invitation creation in controllers/overall_dashboard.js must verify the actor controls every selected hotel, store the short-code invitation payload, and return only the generated code/token metadata needed by the frontend.
- Financial actions
  - Wallet, commission, payout, expense, and financial report actions must remain restricted to owner/admin/finance/reservation roles with valid hotel/account scope.
  - Approved order-taker/agent accounts may read overall financial surfaces only for their assigned hotels and their own agent wallet/reservation records. If financial-actions is opened by an agent, both reservation actions and pending wallet-claim rows must be filtered to that actor's own agent id.
  - Overall financial-actions returns both reservation finance actions and a separately paginated pending wallet-approval queue. Pending wallet claims are sourced from AgentWallet records with source=agent_claim, status=pending, and reviewStatus=pending.
  - Overall financial-report wallet movements must expose financialStatus and reconciliationEligible so rejected claims can remain visible while staying excluded from wallet totals and reconciliation. Rejected rows are audit history only, not reservation reconciliation data.
  - Agent wallet summaries must return both commissionPaid and commissionDue totals. commissionPaid is the already-paid-out commission amount, while commissionDue is the unpaid/outstanding amount. Frontend reports and exports depend on this split for agent and finance reconciliation.
- Agent overall regression scenario
  - Create/approve an order-taker agent assigned to multiple hotels, then create reservations for that agent and for another agent in the same hotel set.
  - As the approved agent, GET /overall-dashboard/pending-reservations/:userId with ownerId and optional assigned hotelId. The response must include only the authenticated agent's pending/rejected/finance-review/commission-review reservations and must exclude every other agent's records.
  - As the approved agent, GET /overall-dashboard/financial-actions/:userId and GET /agent-wallet/summary/:hotelId/:userId, including attempts to pass another agentId. The backend must still return only the authenticated agent's wallet, commission, reservation, and claim data.
  - Repeat with an unassigned hotelId. The API must not expose data for that hotel.
  - Repeat as reservation employee, finance, manager, Hotel System Admin, and super admin to confirm staff/admin queue visibility still follows their broader role scope and was not narrowed by the agent-only filters.
- Public website routes
  - Keep public Janat routes separate from authenticated hotel-management behavior. Public booking/payment endpoints still need reservation/payment validation.

2026-05-25 Implementation Log
- Role naming and access language
  - The hotel-side systemadmin role is now documented and displayed as Hotel System Admin.
  - Keep the stored/internal roleDescription value as systemadmin and the numeric role 10000 for compatibility.
  - Hotel System Admin is scoped hotel-side administration, not platform-wide administration.
  - Platform admins and true super admins remain separate: super admins are global platform owners, while Hotel System Admin users are limited to assigned hotels.
- Overall executive reports for hotel owners and authorized hotel staff
  - Executive summary data must use owner/hotel scoped backend routes instead of reusing global admin report routes directly.
  - Super admins can view across the platform; owners, Hotel System Admins, hotel managers, finance users, reservation employees, and other permitted roles must be constrained to assigned hotels.
  - Use getAccessibleOverallHotels, requireOverallSection, and related overall-dashboard helpers before applying hotelId, invHotel, ownerId, or query filters.
  - Never trust ownerId or hotelId from the query string until it has been checked against the acting user's accessible hotel set.
  - The overall summary page has four business surfaces: general hotel summary, reservation analytics, inventory/occupancy, and paid reservations overview.
  - Admin global reports remain separate from hotel-management executive reports for security and data-boundary clarity.
- Reservation executive report backend behavior
  - The reservation analytics endpoint supports hotel, status, source, payment, search, dateBy, dateFrom, dateTo, includeCancelled, and excludeCancelled query semantics.
  - dateBy must map only to approved date fields such as createdAt, check-in/arrival, or checkout/departure.
  - Cancelled, no-show, and pending reservations must be counted only when the filter or include/exclude setting says so. Do not silently exclude them when a user explicitly asks for those statuses.
  - Aggregations should return creation-date, arrival-date, departure-date, status, source, payment, hotel, and room-type level summaries needed by the frontend charts.
  - Status labels may be humanized on the frontend, but backend values should remain normalized for filtering and reporting stability.
- Inventory/occupancy executive report backend behavior
  - The hotel-management overall inventory report must require one selected hotel before returning occupancy output.
  - Occupancy is calculated from arrival/check-in dates, not creation date or checkout date.
  - The month/range should be controlled by calendar type plus month/year selectors. Frontend may pass invCal, invHMonth/invHYear, invStart, and invEnd, but backend must compute and validate the effective range consistently.
  - Hijri is the default report calendar and should default to the current Hijri month when no valid month query exists.
  - includeCancelled/excludeCancelled must be honored consistently so the UI toggle matches the returned data.
  - The overall inventory endpoint should mirror the single-hotel inventory report structure while still using overall role and hotel-scope validation.
  - Avoid response/query feedback loops: do not mutate query dates from backend responses in a way that causes alternating invHMonth/invStart/invEnd requests.
- Paid reservations overview backend behavior
  - The paid overview returns hotel-level breakdown first, followed by the rest of the report data.
  - Payment summaries must distinguish paid amount, due amount, commission, and booking source/hotel grouping.
  - Data must stay scoped to assigned hotels unless the actor is a true super admin.
- Reservation summary modal and export tracking
  - Reservation list summary exports must create ActivityTracker records so report downloads are attributable.
  - Tracked metadata should include actor, owner scope, hotel scope, active filters, selected date field, row counts, and export/report type.
  - Summary aggregations include date, booking source, status, and room type. Source/status/room-type summaries should be sorted by total amount descending.
- B2B chat backend
  - Chat is B2B only and lives under the b2b-chat backend controller/routes/model/service structure.
  - Staff can chat internally with users allowed by their assigned hotel scope.
  - Hotel managers, Hotel System Admins, owners, finance users, and reservation users can chat with agents assigned to their hotels only.
  - Reception and housekeeping-style accounts should not gain agent chat access unless their role/scope explicitly permits it.
  - Owners and Hotel System Admins are powerful inside their assigned hotels but are not global chat users.
  - Super admins can view and participate across the platform.
  - Chat recipient lookup must filter by role and shared hotel assignment. A hotel owner should not see unrelated super admins or unrelated agents as start-new-chat recipients.
  - Chat recipient API responses should provide friendly roleLabel values such as Owner, Hotel System Admin, Hotel Manager, Finance, Reservations Officer, Agent, and Super Admin. Do not return numeric role codes as the display fallback.
  - Active chats are not closed chats. Historical chats are closed chats.
  - Chats with no activity for 2 hours should be marked seen and closed by the maintenance service. Shorter timeouts may be used only for local testing and then reverted.
  - Attachments and pasted screenshots are supported. Backend responses should expose enough metadata for inline preview while keeping safe download/open URLs available.
  - Unread counters and seen updates must not create request loops. Prefer stable polling/socket triggers and avoid marking a chat as seen merely because the top-navbar icon was clicked.
  - Socket events should refresh active chat lists and unread counters without rebinding listeners on every render.
  - Socket typing events use b2bTyping and b2bStopTyping rooms scoped to b2b-chat:<chatId>. They are transient relays only and should not write to MongoDB.
  - Frontend chat pages may receive legacy/null attachment rows from historical data. Backend should continue returning safe attachment metadata, but frontend must also guard null attachment entries before previewing kind/type/source fields.
  - The B2B chat route is an overall hotel-management surface, not a single-hotel surface. Top-navbar notification context should stay owner/overall scoped unless the user opens a true single-hotel route.
- Executive summary filter model
  - The overview, reservations overview, and paid reservations overview executive tabs may accept multi-hotel and multi-month filters for broad historical analysis.
  - These non-inventory executive tabs may send hotelId/invHotel as comma-separated lists, invMonths/invHMonth/invHYear/invCal, dateFrom/dateTo, invStart/invEnd, dateBy, status, search, and includeCancelled.
  - Backend report handlers must keep parsing repeated or comma-separated hotel/status/month params while enforcing accessible-hotel scope before aggregation.
  - Inventory remains stricter than the other executive tabs: exactly one hotel must be selected, and occupancy remains based on arrival/check-in date.
- Query, loop, and performance guardrails
  - Overall summary and inventory endpoints are expensive; frontend callers should debounce/stabilize query dependencies and backend should avoid unnecessary broad queries.
  - The summary endpoint should not be repeatedly called by the inventory report unless that tab truly needs summary data.
  - GET request logs alternating between different invHMonth/invStart/invEnd values usually indicate a frontend query synchronization loop, not a database issue.
  - For report work, always verify that selected filters, reflected UI labels, and backend results agree before considering the feature complete.
- 2026-05-25 backend file and route map
  - server.js mounts the backend route modules. Any new route must be registered here or through an already-mounted route file.
  - routes/overall_dashboard.js owns hotel-management overall dashboard/report endpoints.
  - controllers/overall_dashboard.js owns the owner/hotel scoped executive summary logic, reservation analytics, inventory/occupancy report data, paid overview data, account-management actions, and signup invitation handling.
  - controllers/reservations.js remains the single-hotel reservation workflow source and should stay aligned with overall reservation list filtering/sorting behavior.
  - routes/b2b_chat.js and controllers/b2b_chat.js own B2B chat access, recipient lookup, chat lists, messages, attachments, unread counters, seen state, and close-chat behavior.
  - models/b2b_chat.js stores B2B chat threads/messages/attachment metadata.
  - services/b2bChatMaintenance.js owns inactive-chat cleanup. Production inactivity should remain 2 hours after local testing.
  - services/activityTracker.js records account-management mutations and report/export activity, including reservation summary report downloads.
  - services/auditPrivacy.js must continue sanitizing privileged audit actor data before reservation/report responses are returned to non-super-admin viewers.
  - Frontend callers for the overall report routes are mainly hotels_frontend/src/HotelModule/apiAdmin.js and hotels_frontend/src/HotelModule/TheOverallStructure/OverallSummary/*.js.
  - Frontend callers for B2B chat are mainly hotels_frontend/src/HotelModule/B2BChat/* and hotels_frontend/src/HotelModule/AdminNavbar/TopNavbar.js.
  - If future logs show repeated OPTIONS/GET calls for chat or inventory, check frontend effect dependencies first, then verify that backend endpoints are idempotent and do not return query-normalization values that cause URL churn.

Recent Production Guardrails
- Reservation detail privacy, May 2026:
  - Non-super-admin viewers must never receive privileged admin actor names in reservation detail, pending queue, finance queue, report, or public-safe reservation responses.
  - When extending reservation workflow metadata, run the field through services/auditPrivacy.js if it contains a by/actor/admin/lastUpdatedBy style snapshot.
- Payment-state consistency, May 2026:
  - Backend payment fields may contain both a selected payment method and actual collected amounts. Do not infer a paid reservation from a method label alone.
  - Reconciliation and financial reports should use recorded/captured amounts and review statuses, not display labels, when deciding paid/due/rejected/eligible states.
